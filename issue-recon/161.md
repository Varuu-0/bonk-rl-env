# Issue Recon: #161 Worker Pool callback map leaks after msgId counter overflow

## Summary
WorkerPool msgId counter increments forever without wrapping, leading to callback map leak and corruption after 2^53 messages.

## Severity
Minor

## Files / Locations
- `/src/core/worker-pool.ts` lines 216-227
- Specifically: Line 33 (`private msgId = 0;`) and Line 216 (`const id = this.msgId++;`)

## Root Cause
The `msgId` counter increments forever. On timeout, the callback is deleted but the counter never wraps. After 2³² messages it will overflow and reuse existing entries, causing callback corruption. After 2^53 it will lose integer precision completely and produce duplicate message ids.

## Reproduction Path
Occurs automatically after ~9 quadrillion messages, or approximately 2.85 years of continuous runtime at 100k steps/sec. Will happen much sooner in large scale distributed training.

## Suspected Files
- `/src/core/worker-pool.ts` (primary location of the bug)

## Root Cause Candidates
1. Missing modulo operation on msgId counter
2. No message ID recycling mechanism
3. No bounds checking or reset mechanism for the counter

## Ambiguity Risks
- Low risk: The issue is well-defined with clear evidence and suggested fixes
- The fix needs to consider performance impact of modulo operation vs periodic reset
- Need to ensure backward compatibility with existing message ID expectations

## Suggested Fix
Use modulo on msgId, or reset counter periodically when the map is empty. Implement proper message id recycling.

## Related Follow-Up
- Add test for msgId wrapping behavior
- Add validation for duplicate message ids
- Add warning when approaching integer precision limit
