# Analysis Bead: Worker Pool msgId Counter Overflow

## Issue Summary
WorkerPool msgId counter increments forever without wrapping, leading to callback map leak and corruption after 2^53 messages.

## Files / Locations
- `/src/core/worker-pool.ts` lines 216-227

## Root Cause Analysis
The `msgId` counter (line 33) increments forever with `this.msgId++` (line 216) without any wrapping mechanism. When a message times out, the callback is deleted from the map (line 221) but the counter continues to increase. This leads to:
1. **Callback map leak**: The map keeps growing with entries that are never reused
2. **Integer precision loss**: After 2^53 messages (~9 quadrillion), the counter loses integer precision and produces duplicate IDs
3. **Callback corruption**: Duplicate IDs cause incorrect callback resolution, leading to silent misrouting

## Failure Mechanism
1. Each message gets a unique incrementing ID: 0, 1, 2, 3, ...
2. On timeout or completion, the callback is removed from the map but ID is never reused
3. Map grows linearly with number of messages processed
4. After 2^53 messages, floating point precision limits cause ID duplication
5. Duplicate IDs cause wrong callbacks to be resolved, breaking message-response pairing

## Fix Scope
Modify the `sendMessage` method in `WorkerPool` class to implement message ID recycling:
- Option 1: Use modulo arithmetic to wrap IDs at a safe threshold
- Option 2: Reset counter when callback map is empty
- Option 3: Implement free ID list to reuse IDs from completed/cancelled messages

## Required Tests
1. **Unit test for ID wrapping behavior**: Verify that IDs wrap correctly after reaching threshold
2. **Integration test for long-running scenarios**: Simulate many messages to ensure no map growth
3. **Duplicate ID detection test**: Ensure no duplicate IDs are generated within a reasonable timeframe
4. **Callback correctness test**: Verify that responses always match the correct requests even after ID wrapping
5. **Stress test**: High-frequency messaging to validate stability under load
